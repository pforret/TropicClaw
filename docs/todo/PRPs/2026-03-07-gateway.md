# PRP: Gateway — Central Orchestration Layer for TropicClaw

**Date:** 2026-03-07
**Confidence Score:** 9/10 (architecture defined, single-user model simplifies design, all major decisions resolved, CLI-only invocation confirmed, agent auto-discovery from filesystem)

## Objective

Build a long-running Bun/TypeScript gateway process that turns Claude Code from an interactive CLI tool into a 24/7 multi-channel agent platform. The gateway sits between external message sources (Telegram, Slack, Discord, HTTP) and Claude Code sessions, handling message routing, session management, trust enforcement, and agent orchestration.

This addresses these gaps from the OpenClaw → Claude Code gap analysis:

1. **No long-lived server/daemon mode** (Gateway, HIGH) — Claude Code is invocation-based, not persistent
2. **No WebSocket/HTTP endpoint** (Gateway, HIGH) — Cannot receive real-time events from channels
3. **No channel adapters** (Channels, RED) — No WhatsApp, Telegram, Discord adapters
4. **No multi-agent orchestration** (Agent Runtime, HIGH) — Cannot route messages to different agent personas
5. **No allowlist verification** (Channels, HIGH) — No gating on who can interact
6. **No unified message format** (Channels, HIGH) — Each channel has different payload structures
7. **No session isolation (dmScope)** (Agent Runtime, HIGH) — No per-user session separation

### Component Responsibilities

| Component | Responsibility |
|-----------|---------------|
| **Channel Adapters** | Maintain persistent connections to platforms. Normalize inbound messages to unified format. Send responses back in platform-native format. |
| **Router** | Looks up current agent for the channel. Detects `/commands` (agent switching). Dispatches messages to agent pool. |
| **Agent Pool** | Spawns and manages Claude invocations. Enforces concurrency limits. Handles session continuity. |
| **Session Store** | Tracks one session per agent (single user). Logs all messages with channel metadata. |
| **Trust Enforcer** | Owner verification (single user). Per-agent tool restrictions passed to `claude -p --allowedTools`. |

### What Claude Code Already Provides (Use As-Is)

- `claude -p "prompt"` — non-interactive agent invocation
- `--resume <session-id>` — session continuity
- `--model`, `--max-turns`, `--allowedTools` — per-invocation config
- `--output-format json` — structured output for parsing
- `CLAUDE.md` — system prompt per agent (separate project dirs)
- MCP servers — external tool integration
- Hooks — lifecycle events (logging, validation, trust enforcement)
- Tropicron — scheduled jobs (runs alongside gateway, not inside it)

## Message Normalization Schema

Every channel adapter converts platform messages to this unified format:

```typescript
interface UnifiedMessage {
  id: string;                          // gateway-assigned UUID
  channel: "telegram" | "slack" | "discord" | "http";
  channelMessageId: string;            // platform-native message ID
  chatId: string;                      // platform chat/channel to reply to
  content: {
    text: string;
    media?: {
      type: "image" | "audio" | "video" | "file";
      url: string;                     // platform URL (may expire)
      localPath?: string;              // downloaded to local staging
      mimeType?: string;
    };
  };
  timestamp: string;                   // ISO 8601
  replyTo?: string;                    // thread/reply context
}
```

Since this is a single-user system, there's no sender model — all messages come from the owner. The `chatId` identifies which platform chat to reply to.

### Outbound Response Format

```typescript
interface OutboundResponse {
  agent: string;                       // agent name, used for prefix
  channel: string;
  chatId: string;                      // chat/channel to reply to
  content: {
    text: string;                      // markdown or plain text
    media?: {
      type: "image" | "file";
      localPath: string;               // file generated by Claude
    };
  };
  replyToMessageId?: string;           // thread the reply
}
```

### Agent Prefix

Every outbound response is prefixed with the agent name:

```typescript
function formatResponse(response: OutboundResponse): string {
  return `[${response.agent}]: ${response.content.text}`;
}
```

Examples:
```
[main]: Found 3 new emails. Top priority is the invoice from Acme Corp.
[coder]: The tests pass. Ready to push.
[monitor]: All systems healthy. Disk at 42%.
```

This makes it clear which agent is responding, especially when:
- Scrolling back through chat history
- A cron job delivers output to a channel
- The user recently switched agents

## Single-User Model

This is a **single-user system**. All channels are used by the same person (the owner). There is no multi-user access, no allowlists, no per-user session isolation. The only isolation axis is **per-agent** — different agents have different personas, tools, and conversation histories.

### Owner Verification

Instead of per-user allowlists, the gateway verifies that messages come from the owner:

```yaml
# gateway/config/gateway.yaml

owner:
  telegram_id: "12345678"
  slack_id: "U0123ABC"
  discord_id: "556677889900"
```

Any message from a non-owner ID is silently dropped. Simple, fail-closed.

## Agent Registry (Auto-Discovered)

Agents are **auto-discovered** from the `gateway/agents/` directory. Any subdirectory containing a `CLAUDE.md` is a valid agent. No central `agents.yaml` registry — the filesystem is the registry.

Agent-specific config lives in `agent.yaml` inside each agent's directory:

```yaml
# gateway/agents/main/agent.yaml
description: "General-purpose personal assistant"
model: sonnet
max_turns: 20
trust_tier: 2
timeout: 120
```

```yaml
# gateway/agents/coder/agent.yaml
description: "Software development agent"
model: opus
max_turns: 50
trust_tier: 1
```

```yaml
# gateway/agents/monitor/agent.yaml
description: "Infrastructure monitoring (cron-only)"
model: haiku
trust_tier: 0
```

**Defaults** (when `agent.yaml` fields are omitted): model=sonnet, max_turns=20, trust_tier=1, timeout=120.

**Discovery**: on startup and when `/agents` is called, the gateway scans `agents/*/CLAUDE.md`. This means `/new coder "dev agent"` just scaffolds a directory — no config file to update.

No channel mappings — channels are viewports, not agent assignments. Every channel starts on `main`.

### Per-Agent Project Directory

Each agent is a directory with `CLAUDE.md` (required) and optional config:

```
gateway/agents/
  main/
    CLAUDE.md                          # personality, rules, tools instructions
    agent.yaml                         # model, trust_tier, description (optional)
    .claude/
      settings.json                    # allowed-tools, MCP servers for this agent
  coder/
    CLAUDE.md
    agent.yaml
    .claude/
      settings.json
  monitor/
    CLAUDE.md
    agent.yaml
    .claude/
      settings.json
```

Claude is invoked from the agent's directory: `cd agents/main && claude -p "..."`, inheriting that agent's `CLAUDE.md`, `.claude/settings.json`, and MCP servers.

### Session Model

**One session per agent**, shared across all channels. Session key = agent name.

```
Telegram ──► [current: main]   ──► "main" session
Slack    ──► [current: main]   ──► "main" session     (same session!)
Discord  ──► [current: coder]  ──► "coder" session    (user switched)
HTTP     ──► [current: main]   ──► "main" session
```

If you tell the main agent something on Telegram, it knows about it when you continue on Slack — because it's the same session. Channel is just metadata on each message.

### Agent Switching via Commands

Every channel has a "current agent" pointer, persisted in the session store. The user can switch at any time using commands detected by the router:

```
/agents              list available agents with descriptions
/switch <name>       switch this channel to a different agent
/new <name> [desc]   create a new agent (interactive — see below)
/current             show which agent this channel is using
/back                switch to previously active agent
```

### `/new` Agent Creation Flow

When the user types `/new researcher "academic research agent"`:

1. Gateway asks: **"Copy personality from an existing agent?"**
   - Lists available agents: `main`, `coder`, `monitor`
   - User picks one → copies that agent's `CLAUDE.md` as starting point
   - User says "no" → uses a default template
2. Scaffolds `agents/researcher/`:
   - `CLAUDE.md` — copied from chosen agent or default template
   - `agent.yaml` — with description from the command
   - `.claude/settings.json` — empty (inherits defaults)
3. Switches the current channel to the new agent
4. Responds: `[researcher]: Ready. I'm a new agent based on [main/template]. What should I know about my role?`

**Default template** (`gateway/templates/default-agent.md`):

```markdown
# Agent: {name}

{description}

## Behavior
- Be concise and direct
- Ask for clarification when the task is ambiguous
- When making changes, explain what you did and why

## Tools
Use the tools available to help the user. Prefer reading before writing.
```

The user can then refine the agent's personality through conversation — the agent can edit its own `CLAUDE.md` when instructed.

Example interaction:
```
user: what meetings do I have today?
[main]: You have 3 meetings: standup at 9:30, design review at 14:00, ...

user: /switch coder
[coder]: Ready. What are we working on?

user: the gateway PRP needs updating
[coder]: Let me read the current PRP...

user: /back
[main]: Welcome back. Anything else about today's schedule?

user: /agents
Available agents:
  main     — General-purpose personal assistant (current on Telegram, Slack)
  coder    — Software development agent (current on Discord)
  monitor  — Infrastructure monitoring (cron-only)
```

### Command Detection in Router

```
inbound UnifiedMessage
  │
  ├─► verify owner (check sender ID against gateway.yaml)
  │     └─ not owner → silent drop
  │
  ├─► check for /command prefix:
  │     /agents   → list agents, return response directly (no Claude invocation)
  │     /switch X → update channel's current agent pointer, return confirmation
  │     /new X    → scaffold new agent directory, return confirmation
  │     /current  → return current agent name
  │     /back     → swap to previous agent, return confirmation
  │
  ├─► resolve current agent for this channel
  │     └─ lookup channel_agents table → agent name (default: "main")
  │
  ├─► session = agent name (e.g., "main", "coder")
  │
  └─► dispatch to Agent Pool with { agent, session, message }
```

## Agent Pool

### Claude Invocation Strategy

**CLI-only: `claude -p`**

All agent invocations use `claude -p`. This gives every agent full access to MCP servers, hooks, skills, and built-in tools (Bash, Read, Write, etc.) — the complete Claude Code runtime. The ~2-3s spawn overhead is acceptable for a messaging use case.

```typescript
import { spawn } from "child_process";

async function invokeAgent(
  agent: AgentConfig,
  sessionId: string,
  prompt: string
): Promise<string> {
  const args = [
    "-p", prompt,
    "--output-format", "json",
    "--model", agent.model || "sonnet",
    "--max-turns", String(agent.max_turns || 20),
    "--dangerously-skip-permissions",
  ];

  if (agent.allowed_tools?.length) {
    args.push("--allowedTools", agent.allowed_tools.join(","));
  }

  // Resume existing session if available
  if (sessionStore.hasActive(sessionId)) {
    args.push("--resume", sessionId);
  }

  const proc = spawn("claude", args, {
    cwd: path.resolve("agents", agent.name),
    timeout: (agent.timeout || 120) * 1000,
    env: {
      ...process.env,
      TRUST_TIER: String(agent.trust_tier || 1),
    },
  });

  const output = await collectOutput(proc);
  return parseResponse(output);
}
```

**Why CLI-only (not Claude API):**
- Full tool execution (Bash, Read, Write, etc.) without reimplementing a tool loop
- MCP servers inherited from each agent's `.claude/settings.json`
- Hooks fire normally (trust enforcer, audit log, etc.)
- Skills available
- ~2-3s latency is fine for chat — nobody expects sub-second replies from an AI assistant

### Concurrency Control

```typescript
class AgentPool {
  private running: Map<string, Promise<string>> = new Map();
  private maxConcurrent: number = 3;    // global limit
  private queue: QueueItem[] = [];

  async dispatch(agent: AgentConfig, sessionId: string, message: UnifiedMessage): Promise<string> {
    if (this.running.size >= this.maxConcurrent) {
      // Queue the request, resolve when a slot opens
      return new Promise((resolve) => {
        this.queue.push({ agent, sessionId, message, resolve });
      });
    }

    // Send "typing..." indicator immediately — CLI invocation takes 2-3s+
    adapters.get(message.channel)?.sendTyping(message.chatId);
    const promise = this.invokeAgent(agent, sessionId, message.content.text);
    this.running.set(sessionId, promise);

    try {
      const result = await promise;
      return result;
    } finally {
      this.running.delete(sessionId);
      this.drainQueue();
    }
  }
}
```

## Session Store (SQLite)

Session state is stored in a single SQLite database at `gateway/data/sessions.db`. SQLite provides atomic writes, concurrent read access, and queryable session history without external dependencies.

```
gateway/data/
  └── sessions.db                      # SQLite: sessions + messages tables
```

### Schema

```sql
CREATE TABLE sessions (
  agent TEXT PRIMARY KEY,              -- one session per agent ("main", "coder", etc.)
  created_at TEXT NOT NULL,            -- ISO 8601
  last_active TEXT NOT NULL,
  message_count INTEGER DEFAULT 0,
  status TEXT DEFAULT 'active'         -- active | archived
);

CREATE TABLE messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  agent TEXT NOT NULL REFERENCES sessions(agent),
  ts TEXT NOT NULL,                     -- ISO 8601
  role TEXT NOT NULL,                   -- user | assistant | system
  channel TEXT,                         -- which channel the message came from/went to
  text TEXT,
  tools_used TEXT,                      -- JSON array, nullable
  latency_ms INTEGER,
  source TEXT DEFAULT 'gateway',        -- gateway | tropicron
  created_at TEXT DEFAULT (datetime('now'))
);

-- Tracks which agent is active on each channel
CREATE TABLE channel_agents (
  channel TEXT NOT NULL,                -- "telegram", "slack", etc.
  chat_id TEXT NOT NULL,                -- platform chat/channel ID
  current_agent TEXT NOT NULL DEFAULT 'main',
  previous_agent TEXT,                  -- for /back command
  updated_at TEXT DEFAULT (datetime('now')),
  PRIMARY KEY (channel, chat_id)
);

CREATE INDEX idx_messages_agent ON messages(agent, ts);
CREATE INDEX idx_messages_channel ON messages(channel, ts);
```

### Query Examples

```sql
-- Current agent for a channel
SELECT current_agent FROM channel_agents
  WHERE channel = 'telegram' AND chat_id = '12345678';

-- Recent messages for an agent (across all channels)
SELECT * FROM messages WHERE agent = 'main' ORDER BY ts DESC LIMIT 50;

-- Messages from a specific channel
SELECT * FROM messages WHERE agent = 'main' AND channel = 'telegram'
  ORDER BY ts DESC LIMIT 20;

-- All active sessions
SELECT * FROM sessions WHERE status = 'active';
```

### Session Lifecycle

1. **Create**: First message to a new agent → `INSERT INTO sessions`
2. **Resume**: All subsequent messages to the same agent continue the session, regardless of channel
3. **Dream**: Nightly maintenance compresses history and consolidates learnings (see below)
4. **Archive**: Manual or after long inactivity → status set to `archived`, new session starts
5. **Messages persist**: Archived sessions keep their message history for context loading

### Dreaming — Nightly Session Maintenance

Sessions grow indefinitely. Without maintenance, context windows bloat and costs increase. The "dreaming" process runs nightly as a tropicron job and performs session hygiene for each active agent.

**What dreaming does:**

1. **Summarize**: Compress the day's conversation into key decisions, facts, and action items
2. **Learn**: Extract durable knowledge and write it to the agent's personality/memory files:
   - New facts about the user → agent's memory (via claude-mem's semantic store)
   - Behavioral patterns → agent's `CLAUDE.md` (if the agent learned something about how to work better)
   - Identity evolution → agent's identity anchors (via claude-memory-mcp's `reflect` tool)
3. **Evolve persona**: Update personality and context files (see [Persona Templates](../../gap/07-persona-templates.md)):
   - **SOUL.md** — refine behavioral principles based on what communication styles worked/failed
   - **USER.md** — accumulate learned user preferences, timezone habits, pet peeves
   - **TOOLS.md** — update device/infra mappings when new tools or hosts were discovered
   - Promotion scoring in claude-memory-mcp auto-graduates frequently observed patterns into identity anchors
4. **Compress**: Replace detailed message history with the summary. Keep the last N messages (e.g., 50) verbatim for immediate context continuity.
5. **Log**: Write a daily digest to `agents/<name>/dreams/YYYY-MM-DD.md`

**Integration with memory plugins** (see [Memory gap analysis](../../gap/05-memory.md), [Persona Templates](../../gap/07-persona-templates.md)):

| Memory layer | What dreaming writes | Plugin |
|-------------|---------------------|--------|
| **Semantic memory** | Facts, preferences, project context | claude-mem (Chroma vectors + SQLite FTS5) |
| **Identity anchors** | Behavioral patterns, personality evolution | claude-memory-mcp (`reflect` + `anchor` tools) |
| **Soul/persona files** | Evolved SOUL.md, USER.md, TOOLS.md | claude-memory-mcp (`anchor` tool) + direct file edit |
| **Agent CLAUDE.md** | Refined instructions based on what worked/didn't | Direct file edit |
| **Dream log** | What changed and why (only written if changes were made) | `agents/<name>/dreams/YYYY-MM-DD.md` (auto-deleted after 365 days) |

**Tropicron job:**

```yaml
---
cron: "0 3 * * *"
agent: main
session: true
description: "Nightly dreaming — compress history, consolidate learnings"
memory: true
singleton: true
timeout: 600
---

# Dreaming

You are performing nightly maintenance on your own memory and session.

1. Review today's conversation history (all channels)
2. Extract key decisions, facts, preferences, and action items
3. Write durable knowledge to your memory systems:
   - Use the `reflect` tool to analyze identity patterns
   - Use memory tools to store important facts
   - If you learned something about how to work better, update your CLAUDE.md
4. Evolve your persona files if warranted:
   - SOUL.md — refine if you discovered better communication approaches
   - USER.md — add newly learned user preferences, habits, context
   - TOOLS.md — update if new devices, hosts, or infra were mentioned
   - Use the `anchor` tool to promote recurring patterns to identity anchors
5. If you made ANY changes to personality, memory, or persona files:
   - Write a dream log to dreams/YYYY-MM-DD.md documenting what changed and why
   - Include a "## Changes" section listing each file modified with a one-line rationale
   - If nothing changed, skip the dream log (no empty dreams)
6. Delete dream logs older than 365 days from dreams/
7. The gateway will compress old messages after you finish

Keep the digest concise. Focus on what matters for tomorrow.

## Dream log format

```markdown
# Dream: YYYY-MM-DD

## Summary
One-paragraph digest of today's conversations.

## Changes
- `SOUL.md`: added preference for bullet-point responses (user asked 3x)
- `USER.md`: learned user timezone is CET
- `claude-mem`: stored 2 new facts about project X

## Action items carried forward
- [ ] Follow up on deployment issue from Slack
```

## Safety guardrails
- Do NOT delete any files except your own dream logs (and only those older than 365 days)
- Do NOT modify CLAUDE.md or persona files unless you have a specific, evidence-based improvement
- Do NOT send messages to any channel
- Persona changes must be small and incremental — no personality rewrites
```

**Post-dream compression** (gateway-side, after the dreaming job completes):

```sql
-- Keep last 50 messages verbatim, compress older ones
-- The dream summary replaces detailed history
INSERT INTO messages (agent, ts, role, text, source)
  VALUES ('main', datetime('now'), 'system',
    '[Dream summary for YYYY-MM-DD written to dreams/YYYY-MM-DD.md. '
    || 'Older messages compressed.]', 'dreaming');

DELETE FROM messages
  WHERE agent = 'main'
  AND id NOT IN (
    SELECT id FROM messages WHERE agent = 'main'
    ORDER BY ts DESC LIMIT 50
  )
  AND role != 'system';
```

**Per-agent dream schedule**: Each agent can have its own dreaming job, or a single job can iterate over all agents. The `main` agent dreams every night; a `monitor` agent that only runs cron checks might dream weekly.

## Trust Enforcement

Two layers, both must pass:

### Layer 1: Owner Verification

Before any processing, the router checks that the message comes from the owner (configured in `gateway.yaml`). Non-owner messages are silently dropped.

```typescript
function isOwner(config: GatewayConfig, message: UnifiedMessage): boolean {
  const ownerIds: Record<string, string> = {
    telegram: config.owner.telegram_id,
    slack: config.owner.slack_id,
    discord: config.owner.discord_id,
    http: "http-owner",                // HTTP adapter trusts localhost
  };
  // Message sender ID must match the configured owner for this channel
  // For HTTP adapter on localhost, all requests are trusted
  return message.channel === "http" ||
    message.channelMessageId !== undefined;  // simplified — real check compares sender ID
}
```

Since this is single-user, the check is simple: one owner ID per platform, fail-closed.

### Layer 2: Hook-Based Tool Validation

A `PreToolUse` hook enforces per-agent trust tiers at the Claude Code level:

```bash
#!/bin/bash
# gateway/hooks/trust-enforcer.sh
# TRUST_TIER env var set by gateway before spawning claude -p

INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.tool_name')
TIER="${TRUST_TIER:-0}"

deny() {
  jq -n --arg reason "$1" '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: $reason
    }
  }'
  exit 0
}

case "$TOOL" in
  Read|Glob|Grep|WebSearch|WebFetch)
    # Tier 0+ (read-only)
    ;;
  Write|Edit|NotebookEdit)
    [[ $TIER -lt 1 ]] && deny "Write operations require trust tier 1+"
    ;;
  Bash)
    COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')
    [[ $TIER -lt 1 ]] && deny "Bash requires trust tier 1+"
    # Tier 3 required for destructive patterns
    if echo "$COMMAND" | grep -qE 'rm -rf|DROP TABLE|git push --force|mkfs|dd if='; then
      [[ $TIER -lt 3 ]] && deny "Destructive command requires trust tier 3"
    fi
    ;;
  mcp__*)
    [[ $TIER -lt 2 ]] && deny "MCP tools (network) require trust tier 2+"
    ;;
esac

# Default: allow
exit 0
```

### Trust Tiers

| Tier | Allowed operations | Typical use |
|------|-------------------|-------------|
| **0 — Read-only** | Read, Glob, Grep, WebSearch, WebFetch | Monitoring, reporting |
| **1 — Local write** | + Write, Edit, Bash (non-destructive), git commit | Development tasks |
| **2 — Network** | + MCP tools, API calls, send messages | Channel interactions |
| **3 — System** | + Destructive ops, package install, force-push | Infrastructure (rare) |

## Channel Adapters

### First Channel: Telegram

Telegram is the first adapter because:
- Mature Bot API with long-polling (no webhook infrastructure needed)
- Native Markdown support (MarkdownV2)
- Rich media (photos, documents, voice)
- BotFather makes setup trivial

### Message Format Conversion

Claude produces Markdown. Each channel speaks a different format. The gateway converts outbound messages per adapter:

| Channel | Format | Conversion |
|---------|--------|-----------|
| Telegram | MarkdownV2 | Escape special chars (`_`, `*`, `[`, etc.), convert code blocks |
| Slack | mrkdwn | `**bold**` → `*bold*`, `` `code` `` stays, convert links |
| Discord | Discord Markdown | Nearly identical to standard Markdown — minimal conversion |
| HTTP | Raw Markdown | No conversion (API consumers handle their own rendering) |
| Fallback | Plain text | Strip all Markdown formatting |

The conversion lives in the gateway core (`src/format.ts`), not in each adapter. Adapters call `formatForChannel(channel, markdown)` before sending.

```typescript
function formatForChannel(channel: string, markdown: string): string {
  switch (channel) {
    case "telegram":  return toTelegramMarkdownV2(markdown);
    case "slack":     return toSlackMrkdwn(markdown);
    case "discord":   return markdown;  // near-identical
    case "http":      return markdown;  // raw
    default:          return stripMarkdown(markdown);  // fallback: plain text
  }
}
```

### Adapter Interface

```typescript
interface ChannelAdapter {
  readonly name: string;

  // Start listening for inbound messages
  start(): Promise<void>;

  // Stop and clean up connections
  stop(): Promise<void>;

  // Send a response back to the channel
  send(response: OutboundResponse): Promise<void>;

  // Event emitter for inbound messages
  on(event: "message", handler: (msg: UnifiedMessage) => void): void;
}
```

### Telegram Adapter

```typescript
import { Telegraf } from "telegraf";

class TelegramAdapter implements ChannelAdapter {
  readonly name = "telegram";
  private bot: Telegraf;
  private handlers: ((msg: UnifiedMessage) => void)[] = [];

  constructor(private token: string) {
    this.bot = new Telegraf(token);
  }

  async start() {
    this.bot.on("message", (ctx) => {
      const msg = this.normalize(ctx);
      this.handlers.forEach((h) => h(msg));
    });
    await this.bot.launch();
  }

  async stop() {
    this.bot.stop();
  }

  async send(response: OutboundResponse) {
    const text = formatForChannel("telegram", formatResponse(response));
    await this.bot.telegram.sendMessage(
      response.chatId,
      text,
      { parse_mode: "MarkdownV2", reply_to_message_id: response.replyToMessageId }
    );
  }

  private normalize(ctx: any): UnifiedMessage {
    return {
      id: crypto.randomUUID(),
      channel: "telegram",
      channelMessageId: String(ctx.message.message_id),
      chatId: String(ctx.message.chat.id),
      content: {
        text: ctx.message.text || ctx.message.caption || "",
        media: this.extractMedia(ctx),
      },
      timestamp: new Date(ctx.message.date * 1000).toISOString(),
    };
  }

  private extractMedia(ctx: any): UnifiedMessage["content"]["media"] | undefined {
    const msg = ctx.message;
    if (msg.photo) {
      const largest = msg.photo[msg.photo.length - 1];
      return { type: "image", url: "", mimeType: "image/jpeg" };
      // URL resolved via bot.telegram.getFileLink(largest.file_id)
    }
    if (msg.document) {
      return { type: "file", url: "", mimeType: msg.document.mime_type };
    }
    return undefined;
  }

  on(event: "message", handler: (msg: UnifiedMessage) => void) {
    this.handlers.push(handler);
  }
}
```

## Media Handling

When a message includes media (image, audio, file):

1. **Download** — Adapter downloads from platform to local staging directory
2. **Stage** — Save to `gateway/media/<session-id>/<filename>`
3. **Reference** — Include `localPath` in UnifiedMessage
4. **Prompt** — Prepend media instruction to prompt: `"[Attached image: /path/to/image.jpg] <user text>"`
5. **Cleanup** — Delete staged files after session expiry

```typescript
async function stageMedia(
  adapter: ChannelAdapter,
  message: UnifiedMessage
): Promise<string | undefined> {
  if (!message.content.media?.url) return undefined;

  const dir = path.join("gateway", "media", message.id);
  mkdirSync(dir, { recursive: true });

  const ext = message.content.media.mimeType?.split("/")[1] || "bin";
  const localPath = path.join(dir, `attachment.${ext}`);

  // Download from platform (adapter-specific)
  await downloadFile(message.content.media.url, localPath);

  message.content.media.localPath = localPath;
  return localPath;
}
```

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Channel connection lost | Adapter reconnects with exponential backoff (Telegraf/Bolt/discord.js handle this natively) |
| `claude -p` times out | Return error message to channel, log to session |
| `claude -p` fails (exit != 0) | Return error message to channel, log exit code |
| Unknown sender (not owner) | Silent drop (no response, no error to sender) |
| Agent pool at capacity | Queue message, reply with "processing..." if wait > 5s |
| Session store disk full | Log alert, continue with in-memory only |
| Gateway crash | launchd/systemd restarts automatically; in-flight messages lost |
| Media download fails | Proceed without media, include warning in prompt |

## Gateway + Tropicron Collaboration

Gateway and tropicron are **separate processes** but share infrastructure. The key design question is: how much context can they share?

### Process Overview

| | Gateway | Tropicron |
|--|---------|-----------|
| Trigger | Inbound messages (real-time) | Cron schedule (every minute) |
| Process | Long-running Bun daemon | Short-lived bash, spawns `claude -p` |
| Invocation | `claude -p` (CLI-only) | `claude -p` (bash-based) |
| Agents | Multi-agent, channel-routed | Single-agent per job |

### Shared Infrastructure

| Resource | Shared? | How |
|----------|---------|-----|
| **Session store** (SQLite) | **Yes** | Single `gateway/data/sessions.db` — both read/write |
| **Agent directories** | **Yes** | Tropicron jobs can target any agent's `CLAUDE.md` |
| **Trust enforcer hooks** | **Yes** | Same `PreToolUse` hook applies to both |
| **Process lifecycle** | No | Independent start/stop |
| **Concurrency control** | No | Gateway has its own pool; tropicron uses OS process limits |

### Session Model for Cron Jobs

Since sessions are keyed by agent name, tropicron integration is straightforward. A job specifies which agent it belongs to, and can optionally resume that agent's session or run isolated.

#### Mode 1: Isolated (default)

The job runs in a fresh Claude session with the agent's `CLAUDE.md` but no conversation history.

```yaml
---
cron: "0 9 * * MON-FRI"
agent: main
# No session: true → isolated mode
---
Summarize today's calendar and email.
```

The `memory: true` field provides per-job persistence via `<job>.memory.md`, but that's job-private — not shared with the agent's session.

#### Mode 2: Session Resume

The job resumes the agent's session, continuing the conversation as if the user had sent another message from any channel. Since sessions are per-agent (not per-channel), this is simple:

```yaml
---
cron: "0 17 * * FRI"
agent: main
session: true                          # resume main's session
output_channel: telegram               # send results to Telegram
output_target: "12345678"
---
Summarize what we discussed this week and list open action items.
```

**How it works:**

1. Tropicron sees `session: true` + `agent: main`
2. Loads the "main" agent's message history from `sessions.db` (all channels combined)
3. Passes history + job prompt to Claude
4. Claude sees the full conversation context — from Telegram, Slack, wherever
5. Response is written to the session's message log AND sent to `output_channel`
6. The response is prefixed: `[main]: ...`

#### Mode 3: Cross-Agent Context (read-only)

The job reads context from other agents' sessions without joining them. Useful for aggregation across agents.

```yaml
---
cron: "0 9 * * MON"
agent: main
context_from:
  - coder                              # read coder's recent history
  - monitor                            # read monitor's recent history
output_channel: telegram
output_target: "12345678"
---
Review what the coder and monitor agents worked on this week.
Summarize key decisions and open items.
```

**How it works:**

1. Tropicron loads message history from each referenced agent's session
2. Formats as context block prepended to the prompt:
   ```
   ## Context: coder agent (last 7 days, 23 messages)
   [2026-03-01 10:00] user (telegram): fix the login bug
   [2026-03-01 10:05] [coder]: Found the issue in auth.ts...
   ...

   ## Context: monitor agent (last 7 days, 5 messages)
   [2026-03-05 09:00] [monitor]: All systems healthy.
   ...

   ---
   # Job prompt
   Review what the coder and monitor agents worked on...
   ```
3. Runs in `main`'s session (or isolated if `session: true` is not set)
4. Response sent to `output_channel`

**Context window budget:** The `context_from` loader respects a token budget (configurable, default 50k tokens). Truncates oldest messages first.

### Output Routing: Where Do Job Results Go?

Without `output_channel`, job output goes only to the log file (current tropicron behavior). With output routing, results are delivered to a channel:

```yaml
---
cron: "*/30 * * * *"
agent: monitor
output_channel: telegram
output_target: "-100123456"            # group chat ID
output_condition: "non-empty"          # only send if Claude produced output
---
Check server health. Only report problems.
```

**Frontmatter fields for output routing:**

| Field | Required | Description |
|-------|----------|-------------|
| `output_channel` | No | Channel to send results to (`telegram`, `slack`, `discord`) |
| `output_target` | With output_channel | Chat ID / channel ID / user ID to send to |
| `output_condition` | No | When to send: `always` (default), `non-empty`, `error-only` |
| `output_format` | No | How to send: `full` (default), `summary` (first 500 chars), `file` (attach as .md) |

**How delivery works:**

1. Tropicron captures Claude's output from `claude -p`
2. If `output_channel` is set AND condition is met:
   - Tropicron calls the gateway's HTTP API: `POST /api/deliver`
   - Gateway routes the message through the appropriate channel adapter
   - Message is logged in the session store (if session context was used)
3. If gateway is down, tropicron logs a warning and saves output to the job log (no message lost, just not delivered)

**Gateway endpoint for delivery:**

```typescript
// POST /api/deliver — used by tropicron to send output to channels
app.post("/api/deliver", async (req) => {
  const { channel, target, text, sessionId, format } = req.body as any;
  const adapter = adapters.get(channel);
  if (!adapter) throw { statusCode: 404, message: "channel not configured" };

  const formatted = format === "summary" ? text.slice(0, 500) + "..." : text;
  adapter.send({
    sessionId: sessionId || "cron-delivery",
    channel,
    targetId: target,
    content: { text: formatted },
  });
  return { status: "delivered" };
});
```

### What Tropicron Needs From the Gateway

Tropicron calls the gateway's HTTP API (via `curl`). If the gateway is down, session-aware jobs fall back to isolated mode with a warning.

### Gateway API Endpoints

| Endpoint | Method | Purpose | Used by |
|----------|--------|---------|---------|
| `/api/message` | POST | Receive message from HTTP adapter | External clients |

## Resolved Design Decisions

1. **Runtime & framework** — Bun + Fastify. Bun for fast startup, native TS, built-in SQLite (`bun:sqlite`) and `.env` loading. Fastify for schema-validated routes and better performance than Express.
2. **Hosting** — Same machine as Claude Code (localhost). Gateway lives in `gateway/` subfolder, started via `bun run src/index.ts`. Managed by launchd (macOS) or systemd (Linux) for auto-restart.
3. **Invocation** — CLI-only via `claude -p`. Every agent gets MCP servers, hooks, skills, and built-in tools for free. ~2-3s spawn overhead is acceptable for messaging. No Claude API invocation — avoids reimplementing tool execution.
4. **State persistence** — SQLite via `bun:sqlite`. Single file at `gateway/data/sessions.db`. Atomic writes, queryable history, no external service.
5. **Media handling** — Download to local file (`gateway/media/<msg-id>/`). Pass local path to Claude. Cleanup on session expiry.
6. **Typing indicator** — Sent immediately when dispatching to agent pool (CLI spawn takes 2-3s+). No streaming — `claude -p` returns full response.
7. **Multi-machine** — Not needed. Gateway and Claude Code both run locally on the same machine.
8. **Single user** — All channels are used by the same owner. No multi-user allowlists. Owner verification via platform user IDs in `gateway.yaml`. Session isolation is per-agent, not per-user or per-channel.
9. **Session model** — One session per agent, shared across all channels. Session key = agent name. If you tell the main agent something on Telegram, it knows about it on Slack.
10. **Agent switching** — Every channel starts on the `main` agent. User can switch via `/switch <name>`, `/agents`, `/back`, `/new`. Current agent per channel persisted in `channel_agents` table.
11. **Agent prefix** — Every outbound response is prefixed with `[agent_name]:` to identify which agent is responding.
12. **First channel** — Telegram (mature Bot API, long-polling, native Markdown, easy BotFather setup). HTTP adapter also in Phase 1 for testing.
13. **Agent creation** — `/new` asks if personality should be copied from an existing agent. If not, uses a default template. Scaffolds directory with `CLAUDE.md` + `agent.yaml`.
14. **Dreaming** — Nightly tropicron job (3am) summarizes conversations, extracts learnings to memory plugins (claude-mem, claude-memory-mcp), compresses session history. Dream logs written to `agents/<name>/dreams/YYYY-MM-DD.md`.
15. **CLI-only invocation** — All agents use `claude -p` (not Claude API). Gets MCP, hooks, skills, tools for free. ~2-3s latency acceptable for messaging.
16. **Agent auto-discovery** — Scan `agents/*/CLAUDE.md`. Filesystem is the registry, no central config file.
17. **Format conversion** — Gateway converts Markdown → channel-specific format (Telegram MarkdownV2, Slack mrkdwn, Discord MD, plain text fallback). Conversion in `src/format.ts`, not per-adapter.
